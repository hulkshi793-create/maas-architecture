MaaS → DeepSeek 请求体/响应体 逐阶段变化追踪

用户问"什么是MaaS平台？" → 路由到 deepseek-v4-pro → 返回回答 · 10 个阶段

阶段 0
客户端发出
用户通过 MaaS SDK/API 发送请求。携带 MaaS 平台 API Key。
HTTP Request
POST https://maas-gateway.zj.chinamobile.com/v1/chat/completions Authorization: Bearer maas-sk-abc123... Content-Type: application/json { "model": "deepseek-v4-pro", "messages": [ {"role": "system", "content": "你是一个智能助手"}, {"role": "user", "content": "什么是MaaS平台？"} ], "stream": true, "temperature": 0.7, "max_tokens": 4096 }
阶段 1
TLS 终止 + HTTP/2 解析
Envoy 解密 HTTPS，解析 HTTP/2 帧，构建请求上下文。请求体原封不动。
Request Body — 无变化
// 完全不变。Envoy 只做传输层处理，不碰应用层 body。 {"model": "deepseek-v4-pro", "messages": [...], ...}
阶段 2
② 路由匹配
routing-forwarding 插件读取 model 字段，匹配后端路由规则。
model="deepseek-v4-pro" → 目标: external-deepseek 路由组。
Request Body — 无变化
// 插件只读 model 做路由决策，不修改 body。 // 设置 upstream cluster = deepseek_adapter {"model": "deepseek-v4-pro", "messages": [...], ...}
阶段 3
③ 认证鉴权
auth 插件校验 API Key 是否在白名单。
key-status 检查 Key 是否被吊销。
通过则放行，拒绝则返回 401/403。
Request Body — 无变化
// 鉴权只读 Header，不修改 body。 // info-register 注入元数据 Header： X-Service-ID: svc-042 X-Service-Name: maas-agent-platform // Body 不变 {"model": "deepseek-v4-pro", "messages": [...], ...}
阶段 4
④ 限流控制 (×4)
rate-limit → Redis INCRBY QPS
token-rate-limit → 预占 Token 配额
api-key-rate-limit → 该 Key 的请求次数
api-key-token-rate-limit → Key+Token 联合限流
任一超限返回 429。
Request Body — 无变化
// 限流只查 Redis 计数器，不修改 body {"model": "deepseek-v4-pro", "messages": [...], ...}
阶段 5 ★
⑤ Adapter 请求转换
MaaS Adapter 做三件事：
① 替换 API Key（MaaS → DeepSeek）
② 注入 thinking 参数
③ 改 base URL → api.deepseek.com

★ DeepSeek 99% OpenAI 兼容，Body 仅加 1 个字段。
若接百度 ERNIE 则需改字段名+参数范围，更多次变化。
Request Body — 发生变化 ★
POST https://api.deepseek.com/chat/completions ← URL 变了 Authorization: Bearer sk-deepseek-xyz789... ← Key 替换 { "model": "deepseek-v4-pro", "messages": [ {"role": "system", "content": "你是一个智能助手"}, {"role": "user", "content": "什么是MaaS平台？"} ], "stream": true, "temperature": 0.7, "max_tokens": 4096, "thinking": {"type": "enabled"} ← Adapter 注入 }
阶段 6
⑥ DeepSeek 推理 & 返回
DeepSeek V4 Pro 执行推理，生成思维链 + 最终回答。
流式 SSE 返回，每 chunk 含 reasoning_content 和 content。
Response SSE Stream — DeepSeek 原始格式
// Chunk 1: 思维链开始 data: {"choices":[{"delta":{"reasoning_content":"用户询问MaaS平台的定义...","content":""},"index":0}],...} // Chunk 2-N: 更多思维链 data: {"choices":[{"delta":{"reasoning_content":"需要解释三层含义...","content":""},"index":0}],...} // Chunk N+1: 正式回答开始 data: {"choices":[{"delta":{"content":"MaaS（Model as a Service）"},"index":0}],...} // 后续 chunk: 回答续 data: {"choices":[{"delta":{"content":"是指将AI模型..."},"index":0}],...} // 末块: finish + usage data: {"choices":[{"delta":{},"finish_reason":"stop"}],"usage":{"prompt_tokens":42,"completion_tokens":156,"total_tokens":198}} data: [DONE]
阶段 7 ★
⑦ Adapter 响应转换
MaaS Adapter 做两件事：
① 提取 usage → 写入计费事件 (Kafka)
② 流式 chunk 重整：reasoning_content 按配置剥离或保留

★ DeepSeek 响应接近标准 OpenAI，重整量小。
若接千问：末块 choices=[] 需重整为标准格式。
若接百度 ERNIE：result→choices 全量映射。
Response SSE Stream — 经 Adapter 重整（保留 reasoning，提取 usage）
// 计费事件异步发 Kafka: // {request_id, tenant, model, input_tokens:42, output_tokens:156, ...} // Adapter 逐 chunk 处理： // ① 提取 usage（末块）→ 计费 // ② reasoning_content 按配置决定保留/剥离 // ③ 流式透出到客户端 data: {"choices":[{"delta":{"reasoning_content":"用户询问MaaS平台的定义...","content":""},...}],...} data: {"choices":[{"delta":{"content":"MaaS（Model as a Service）"},...}],...} // ... data: {"choices":[{"delta":{},"finish_reason":"stop"}],"usage":{...}} data: [DONE] // ★ 非 DeepSeek 厂商的重整差异： // 千问末块: choices:[], usage:{} → 重整为 choices[{finish:"stop"}], usage:{} // ERNIE: result → choices[0].delta.content 全量映射
阶段 8
⑧ 响应采集
ai-statistics 插件记录：
· TTFT（首 Token 延迟）
· Token 用量（prompt/completion/total）
· HTTP 状态码
· 端到端延迟
Response — 无变化（插件只读不写）
// ai-statistics 只采集指标，不修改响应体。 // 数据走 ALS gRPC → Kafka → ES → Grafana // 采集的指标示例： // TTFT: 320ms, Total Latency: 2.1s // prompt_tokens: 42, completion_tokens: 156 // status: 200
阶段 9
⑨ 客户端接收
用户收到完整的 SSE 流。
包含 reasoning_content（思考过程）+ content（最终回答）。
最终回复："MaaS 是指将 AI 模型作为服务提供..."
Client 收到的 SSE Stream
// 客户端 SDK 解析 SSE 流： // 思考过程（可选展示）： "用户询问MaaS平台的定义，需要解释三层含义..." // 最终回答： "MaaS（Model as a Service）是指将AI模型作为服务提供， 用户无需关心底层GPU、推理框架、模型部署， 通过API即可调用大模型能力。核心价值在于 降低AI使用门槛，按量付费，弹性伸缩。"

🔵 请求体变化

  • · 阶段 0-4: 完全不变（7 个插件只读不改 body）
  • · 阶段 5: Adapter 注入 thinking 参数
  • · Header 变化: MaaS Key → DeepSeek Key
  • · ★ 变化次数取决于厂商：DeepSeek 1 次，百度 ERNIE 2-4 次

🟠 插件只读不改 Body

  • · routing-forwarding: 读 model 做路由
  • · auth/key-status: 读 Header 做鉴权
  • · 4 个限流插件: 读 Redis 做计数
  • · ai-statistics: 读响应做采集
  • · 所有插件操作 Header/Redis，不碰 Body

🟣 响应体重整

  • · DeepSeek: reasoning_content 可选剥离
  • · 千问: 末块 choices=[] 需重整
  • · 百度 ERNIE: result→choices 全量映射
  • · 各厂商兼容度不同，重整量从 0 到全量

🟢 计费链路

  • · 事前: token-rate-limit 预占配额
  • · 事中: Adapter 实时提取 usage
  • · 事后: ALS 日志 → Kafka → ES 对账
  • · 双路: 网关计数 + DeepSeek 账单对账
