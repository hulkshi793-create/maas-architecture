Envoy Proxy Architecture

Envoy 核心组件架构

从 Listener 接收下游请求，到 HTTP Filter Chain 执行，再到 Cluster 管理上游 Endpoint，完整展示 Envoy 的数据流转路径、扩展机制与进程边界。

LDS / RDS / CDS / EDS / SDS
事件驱动 · 多 Worker · 非阻塞 I/O
Wasm 扩展 · 热更新 · 沙箱隔离
01

Listener

监听指定 IP:Port，接收下游连接并完成 TLS 终止。证书通过 SDS 动态下发，请求解密后流量进入 Network Filter。

接入层 · TLS · Downstream
02

Network Filter

由 HTTP Connection Manager（HCM）负责协议解析、HTTP/2 多路复用与 gRPC 支持，并驱动 HTTP Filter Chain 执行。

HCM · HTTP/2 · gRPC
03

HTTP Filter Chain

请求方向按顺序执行过滤器；响应返回时反向再次执行。支持 Wasm 扩展、限流、鉴权、统计与路由转发等能力。

04

Cluster Manager

负责负载均衡、后台健康检查、连接池与原生熔断机制。每个 Worker 维护独立连接池，实现无锁高性能调度。

Load Balance · Circuit Breaker
Envoy Process Boundary
05

Endpoint（Upstream）

实际上游实例，例如 vLLM Pod、厂商 API 或业务服务实例。Endpoint 位于 Envoy 进程外，并通过 EDS 动态同步。

Listener · 接入层

负责监听端口、建立连接、TLS 终止与下游流量接入，是 Envoy 的入口层。

Network Filter · 协议层

HCM 管理 HTTP 生命周期、协议解析、路由触发与 HTTP Filter Chain 执行。

HTTP Filter Chain · 扩展层

支持鉴权、限流、统计、Header 转换等能力，可通过 Wasm 动态扩展。

Cluster / Endpoint · 上游层

Cluster 管理负载均衡与连接池；Endpoint 表示真正的上游实例与服务节点。

Envoy Core Architecture · Optimized Layout Design
Listener → Network Filter → HTTP Filter Chain → Cluster → Endpoint
