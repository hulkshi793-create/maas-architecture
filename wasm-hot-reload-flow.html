Wasm 插件热更新全流程

改 Go 代码 → 编译 → 推送镜像 → 改 CRD → xDS 下发 → Envoy 热加载 · 全程零中断

各阶段说明

阶段一：开发者本地

  • · TinyGo 是 Wasi 目标编译器（不是 go build），产出纯 .wasm 二进制
  • · OCI 镜像用 FROM scratch，只有 .wasm 一个文件，通常 200-500KB
  • · tag v2 就是版本控制——回滚就是 kubectl apply 旧 tag 的 CRD

阶段二：K8s 下发

  • · WasmPlugin 是 CRD（Custom Resource Definition），不是 K8s 原生资源
  • · Controller 用 Informer/Watch 机制监听变更，实时触发 reconcile
  • · 控制器崩溃 = 新配置下不去，但数据面存量不受影响

阶段三：控制面

  • · Pilot 不推全量，只推版本号变化的资源（Delta xDS）
  • · 改 1 条路由只推 1 条，不会因为改了 auth 而把 routing-forwarding 也重推一遍
  • · Pilot 宕机 = 新配置停发，但 Envoy 继续用当前配置

阶段四：Envoy 热加载

  • · Envoy 不重启进程，只创建新的 Wasm 实例并注入 Filter Chain
  • · 新旧实例短时间内共存，旧的处理完 in-flight 请求后销毁
  • · 只改 pluginConfig 的话，不需要拉新 .wasm，调旧实例的 proxy_on_configure() 即可
  • · 引用计数机制保证旧实例安全销毁——不会出现"旧请求还在跑，实例已销毁"的情况

Wasm 插件热更新 · kubectl apply → xDS Delta → Envoy 零中断替换 · 全程无 restart
